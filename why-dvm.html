The Best from Both Worlds: VM and Container

Performance

When VMs take tens of seconds to boot, DVM is able to launch in sub-second. Also, DVM asks the minimal resource footprint: ~10MB mem. You may run hundreds of DVM instances on a server, where a couple tens of VMs will overload.

Secure

DVM is immune from the "shared kernel" problem in container. Virtualization offers an excellent hardware-enforced isolation. The attack surface for a VM instance is quite small, as it lacks the variety of functions (and, therefore, the potential flaws to be exploited) provided by standard operating systems.

Portable

DVM is Hypervisor Agnostic. The current implementation supports KVM and Xen, with more in the roadmap. Combined with the portability of App Container Image, DVM allows you to build, ship, run app anywhere, without concerning the underlying technologies.

Immutable

DVM eliminates the need for guest OS. There are no moving parts inside of a DVM instance to be configured or managed. Therefore the entire stack is immutable.

BYOK - bring your own kernel

In a multi-tenant environment, the platform must allow developers to pick different the kernel and its modules. This is an easy job in DVM, but very hard to do in container, due to the nature of ‚Äùsharing the host kernel".

Production Ready

Virtualization is mature. Features like LiveMigration, SDN, SDS have been battle-tested for years. With DVM, just plug-&-play. You don't have to wait another two years for the container-version of SDN.

Better ROI

Virtualization is commonly implemented among enterprises. It is not surprising that someone has a virtualized infrastructure running. DVM provides a plug-&-play path for seamless migration, instead of rebuilding everything with container.

The following table gives a more detailed comparision between Container, (traditional) VM and DVM:

- Container VM DVM
Isolation Weak, shared kernel Strong, HW-enforced Strong, HW-enforced
Portable Yes, but kernel dependent sometimes No, hypervisor dependent Yes, hypervisor agnostic and portable image
Boot Fast, sub-second Slow, tens of seconds Fast, sub-second
Performance Great OK Good, minimal resource footprint and overhead
Immutable Yes No, configuration management required Yes, only kernel+image
Image Size Small, MBs Big, GBs Small, MBs
Compatibility No, need new tools Great, everything just works Good, it is still a "Machine", much less changes
Mature Not yet Production ready, SDN, SDS, LiveMigration, etc. Yes, just plug-&-play
ROI Rebuild everything with container - Reuse your virtual infrastructure
