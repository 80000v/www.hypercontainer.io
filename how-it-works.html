WHY DVM?

DVM is comprised of two components:

  • DVM Daemon (with REST APIs)
  • DVM CLI

On a physical Linux host:

[root@user ~:]# docker pull nginx:latest
[root@user ~:]# dvm run nginx:latest

A new VM instance is created, and DVM will mount the image from the host onto the instance:

[root@user ~:]# docker ps
[root@user ~:]#
[root@user ~:]# dvm list
xxxxxx

Inside of the VM instance, a minimalist Linux kernel (called HyperKernel) is booted. The kernel employs a tiny Init service, called HyperD, which acts as Pid 1 process to launch apps from the loaded image.

What is Pod

Pod is a concept originated from Google. The key idea behind Pod is that in a microservie architecture usually involves some "helper" programs, such like log, monitoring, cron, etc. These helper programs are built to work co-operatively with the app. Therefore, instead of running in multiple isolated containers, these processes should share the same environment, although they are packaged in different images.

Pod is the first class citizen in DVM

In DVM, a pod consists of a group of app images, launched in a single instance.

[root@user ~:]# dvm run -p nginx rails logstash cronjob

Inside of a instance, different applications from different imagesshare the namespace: PID, Network, IPC, UTS, User. This model provides a familiar view of a tranditional OS, instead of one process per container philosophy:

  • processes can see each other
  • processes can use all IPC facility to communicate
  • processes share a hostname
  • processes have the access to all NICs attahed with the instance, and share the port range
  • processes have the access to all disk attached with the instance

The exception is Mount. Since Pod includes multiple app images, DVM applies Mount namespace to separate the root filesystem between application.
